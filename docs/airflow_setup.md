# Airflow Setup

## Overview

Apache Airflow orchestrates the AdventureWorks data engineering pipeline. It controls the execution order of the local ingestion process and the Databricks Bronze, Silver, Gold, and SQL Analytics stages.

The Airflow environment is kept separate from the main project environment.

---

## 1. Airflow Environment

Create a dedicated Airflow virtual environment:

```bash
python3 -m venv airflow_venv
```

Activate it:

```bash
source airflow_venv/bin/activate
```

Install the Airflow dependencies:

```bash
pip install -r requirements-airflow.txt
```

The Airflow requirements file contains:

```txt
apache-airflow
apache-airflow-providers-standard
```

---

## 2. Airflow Home Directory

Set the Airflow home directory:

```bash
export AIRFLOW_HOME=~/airflow
```

Confirm the value:

```bash
echo $AIRFLOW_HOME
```

Airflow stores its configuration, logs, database, and DAG files inside this directory.

Example:

```text
~/airflow/
├── airflow.cfg
├── airflow.db
├── dags/
├── logs/
└── plugins/
```

---

## 3. Add the DAG

Copy the AdventureWorks DAG into the Airflow DAG directory:

```bash
cp airflow/adventureworks_pipeline.py ~/airflow/dags/
```

The final location should be:

```text
~/airflow/dags/adventureworks_pipeline.py
```

Confirm that Airflow detects the DAG:

```bash
airflow dags list
```

The AdventureWorks DAG ID should appear in the results.

---

## 4. Configure Project Paths

The DAG must reference the correct location of the AdventureWorks project.

Example project path in WSL:

```text
/mnt/c/Users/your-user/Documents/Portfolio/AdventureWorks-End-to-End-Data-Engineering
```

The ingestion task should execute the local pipeline from the `ingestion` directory:

```bash
cd /path/to/project/ingestion
python3 run_pipeline.py
```

Verify that the path configured in the DAG matches the actual location of the repository.

---

## 5. Configure Credentials

The pipeline requires AWS and Databricks credentials.

Configure them as environment variables or load them from a local `.env` file.

Example:

```env
AWS_ACCESS_KEY_ID=your_aws_access_key
AWS_SECRET_ACCESS_KEY=your_aws_secret_key
AWS_DEFAULT_REGION=your_aws_region
S3_BUCKET_NAME=your_bucket_name

DATABRICKS_HOST=https://your-workspace.cloud.databricks.com
DATABRICKS_TOKEN=your_databricks_token
```

Do not commit credentials or `.env` files to GitHub.

The Airflow process must have access to these variables when the DAG runs.

---

## 6. Start Airflow

For a local development environment, start Airflow with:

```bash
airflow standalone
```

This starts the Airflow database, scheduler, API server, and web interface.

Open the Airflow interface in a browser:

```text
http://localhost:8080
```

The terminal displays the username and password generated by Airflow.

---

## 7. Run the Pipeline

In the Airflow web interface:

1. Locate the AdventureWorks DAG.
2. Unpause the DAG.
3. Select **Trigger DAG**.
4. Open the Grid or Graph view to monitor execution.

The expected task order is:

```text
Local Ingestion
      │
      ▼
Bronze Layer
      │
      ▼
Silver Layer
      │
      ▼
Gold Layer
      │
      ▼
SQL Analytics
```

A downstream task executes only when its upstream task completes successfully.

---

## 8. Verify the DAG

You can inspect the DAG from the terminal:

```bash
airflow dags list
```

List its tasks:

```bash
airflow tasks list adventureworks_full_pipeline
```

Check recent DAG runs:

```bash
airflow dags list-runs adventureworks_full_pipeline
```

A successful run should show:

```text
state: success
```

You can also test an individual task:

```bash
airflow tasks test adventureworks_full_pipeline <task_id> <logical_date>
```

Replace `<task_id>` and `<logical_date>` with the task and date you want to test.

---

## 9. Monitoring and Logs

Airflow provides logs for every task execution.

To review a failure:

1. Open the failed task in the Airflow interface.
2. Select **Logs**.
3. Find the Python exception, shell error, or Databricks job error.
4. Correct the issue.
5. Clear or rerun the failed task.

Airflow logs are stored under:

```text
$AIRFLOW_HOME/logs/
```

---

## 10. Common Issues

### DAG does not appear

Check that the DAG file is inside:

```text
$AIRFLOW_HOME/dags/
```

Then run:

```bash
airflow dags list
```

Review DAG processor logs for import errors.

### Python module not found

Confirm that the task changes to the correct project directory before running:

```bash
python3 run_pipeline.py
```

Also confirm that the required virtual environment is active or explicitly referenced by the DAG.

### Credentials are unavailable

Confirm that the Airflow process can access the required AWS and Databricks environment variables.

### Databricks task fails

Review the Airflow task log and the corresponding Databricks job output.

### Downstream tasks do not run

This usually means an upstream task failed. Airflow prevents later stages from running when dependencies are not satisfied.

---

## 11. Successful Setup Criteria

The Airflow setup is working correctly when:

- The AdventureWorks DAG appears in the Airflow interface.
- The DAG can be triggered manually.
- The ingestion task executes `run_pipeline.py`.
- Databricks stages run in the expected order.
- Failed tasks stop downstream execution.
- Task logs are available in Airflow.
- A complete DAG run finishes with the `success` state.
